# data-process

Cleans up [`data-scrape`](../data-scrape/README.md)'s raw
`pokedex.csv` + `images/` into one standardized dataset meant to be
consumed by a future web page: a single JSON file with consistent field
casing and units, and a folder of uniformly-sized artwork images.

## The idea

`data-scrape/compile_pokedex.py` merges two independent sources
(PokeAPI + a pokemondb.net scrape) per Pokemon, and deliberately lets
either source fail without dropping the row - the right call for
*collecting* data (see that stage's README), but it leaves real
inconsistencies in the raw CSV that a web page shouldn't have to work
around:

- **Mixed casing** - types/abilities are normally PokeAPI's lowercase,
  hyphenated slugs (`"solar-power"`), but fall back to pokemondb.net's
  already-human-readable text (`"Cursed Body"`) on any row where
  PokeAPI's request failed. Same data, two different formats depending
  on which source happened to answer for that row.
- **Missing fields with a derivable substitute** - those same rows are
  missing PokeAPI's `height_dm`/`weight_hg`/`sprite_url`, but
  `height_m`/`weight_kg` (from pokemondb.net) are one unit conversion
  away, and a sprite URL follows a predictable pattern from the national
  number.
- **Two unit systems** - the raw CSV keeps both PokeAPI's decimetres/
  hectograms and pokemondb.net's metres/kilograms side by side on
  purpose, as a comparison point in that stage. A web page needs to pick
  exactly one.
- **Inconsistent image sizes** - pokemondb.net's official artwork is
  whatever width x height that Pokemon's actual artwork happens to be
  (checked a sample while building this stage: anywhere from 179 to 400
  pixels tall). A web page laying these out in a grid wants one
  consistent size.

`process_pokedex.py` fixes all four. Run it any time after
`data-scrape/compile_pokedex.py` has produced at least some rows - it
doesn't need the full Pokedex to be done, and is safe to re-run later to
pick up more.

## Files

- `process_pokedex.py`:
  - `humanize()` - normalizes a type/ability string to one consistent
    `"Title Case"` form, whether it came in as a PokeAPI slug or
    already-human-readable pokemondb.net text.
  - `clean_row()` - turns one raw CSV row into one standardized dict, or
    `None` if the row isn't usable web material (no artwork on disk, or
    a field with no source and no derivable substitute). Returns `None`
    rather than raising, so one bad row doesn't stop the rest of the
    dataset from being produced.
  - `standardize_image()` - resizes one artwork image to fit within
    `THUMBNAIL_SIZE` (300x300 by default) and pads it onto a white
    square of exactly that size, so every image in the output is the
    same dimensions regardless of the source artwork's own aspect ratio.
  - `process_pokedex()` - entry point: reads the CSV, cleans + writes
    every row, standardizes every image, and prints a summary of how
    many rows were skipped and why.
- `tests/test_process_pokedex.py` - tests `humanize()` and `clean_row()`
  against synthetic rows shaped like `data-scrape/pokedex.csv` (including
  one shaped like the real rows where PokeAPI failed), not the live CSV
  - fast, and independent of how far `compile_pokedex.py` has progressed.
- `conftest.py` - empty; lets `tests/` import `process_pokedex` without a
  package (same reasoning as `data-scrape/conftest.py`).
- `pokedex.json`, `images/` - generated by running `process_pokedex.py`.
  Unlike this repo's other generated output, these **are** committed
  (see root `.gitignore`): [`data-serve`](../data-serve/README.md) (and
  through it, [`d3-interactive-web`](../d3-interactive-web/README.md))
  needs them to run at all, and reproducing them means re-running
  `data-scrape`'s live scrape against PokeAPI/pokemondb.net - slow, and
  dependent on those sites staying reachable and unchanged. Committing
  this stage's output means a fresh clone of this repo can run
  data-serve immediately, no scrape required. Still fully reproducible
  by running `process_pokedex.py` again if you ever need to (e.g. after
  a `data-scrape` change).

## The output shape

Each entry in `pokedex.json`:

```json
{
  "national_number": 1,
  "name": "bulbasaur",
  "display_name": "Bulbasaur",
  "generation": 1,
  "species": "Seed Pokémon",
  "types": ["Grass", "Poison"],
  "abilities": ["Overgrow", "Chlorophyll"],
  "hidden_ability": "Chlorophyll",
  "height_m": 0.7,
  "weight_kg": 6.9,
  "sprite_url": "https://raw.githubusercontent.com/PokeAPI/sprites/master/sprites/pokemon/1.png",
  "artwork": "images/bulbasaur.jpg"
}
```

Compared to the raw CSV row this came from: `types`/`abilities` are real
JSON arrays (not `;`-joined strings) in one consistent casing;
`height_dm`/`weight_hg` are gone in favor of `height_m`/`weight_kg`
alone; `artwork_url` (the remote pokemondb.net URL) is gone in favor of
`artwork` (the local, standardized image's relative path); and
`fetch_issues`/`artwork_path` (data-scrape's own bookkeeping columns)
are dropped entirely - none of that is something a web page needs.

`name` stays a lowercase slug (matches pokemondb.net/PokeAPI naming,
good as a stable id or URL fragment); `display_name` is the Title Case
version for showing on the page.

## Notes on skipped rows

A row is left out of `pokedex.json` (and logged as skipped in the
summary line) if:
- its artwork file isn't actually present on disk under `data-scrape/`
  (e.g. `compile_pokedex.py` hasn't reached it yet, or its download
  failed), or
- both sources failed to provide height/weight, with nothing to derive
  a value from.

Running it against the completed 721-species CSV, zero rows were
skipped and zero hidden-ability mismatches were flagged - the two
sources agree cleanly once casing/units are normalized. Only 2 of the
721 rows (Gengar, Starly) ever needed the height/weight/sprite-URL
backfill described above, both due to a transient PokeAPI connection
error during the original scrape.

## Running

Needs [Pillow](https://pillow.readthedocs.io/), already in this repo's
`requirements.in`/`.txt` (added for `data-scrape/pokemon_scrapy`'s
`ImagesPipeline`) - no new dependency for this stage.

From the repo root with `.venv` active:

```bash
python data-process/process_pokedex.py
```

Requires `data-scrape/pokedex.csv` to already exist (run
`data-scrape/compile_pokedex.py` first - see that stage's README for
timing).

## Testing

```bash
pytest data-process/tests
```
